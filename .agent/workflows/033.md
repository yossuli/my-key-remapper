# [Workflow-033] 思考シミュレーションによるドキュメント検証 (Thought Simulation Verification)

## 概要

ドキュメント（仕様書やマニュアル）の網羅性を検証するために、**「仕様書を見ずに」ユーザー視点でユースケースを大量に発想し、現状の実装およびドキュメントとのギャップ（乖離）を分析する手法** です。作成者バイアス（「書いたから分かる」）を排除し、ユーザーが本当に知りたい情報が含まれているかを確認します。

## 目的 (Purpose)

- **網羅性の向上**: 作成者が想定していなかった利用シーンを洗い出す。
- **ユーザビリティの確認**: 機能があっても「使い方が分からない（ドキュメントにない）」状態を防ぐ。
- **隠れた機能の発見**: 実装上は可能なのに、知られていない（ドキュメント化されていない）機能を発見する。

## 手順 (Step-by-Step)

### 1. 仕様書の遮断 (Blind Idea Generation)

まず、検証対象のドキュメント（`REMAP_SPECIFICATIONS.md` など）を**閉じます（参照禁止）**。
既存の記述に引っ張られないようにするためです。

### 2. ユースケースの発散 (Brainstorming)

ユーザーになりきって、「このプロダクトでやりたいこと」「ありそうな設定」を大量に列挙します。
「できるかどうか」は一旦無視して、願望ベースで書き出します。

- **視点の例**:
  - 「特定のキー操作でアプリを起動したい」
  - 「マウス操作も自動化したい」
  - 「ゲーム用に連射したい」
  - 「仕事中に誤爆を防ぎたい」
  - 「他の人には使えないようにしたい」

### 3. 事実との突き合わせ (Fact Checking)

洗い出した事例（ユースケース）ひとつひとつに対し、以下の 2 点を確認します。

1.  **実装検証**: 現在のコードベースで技術的に可能か？
    - コード (`src/` 配下) を調査して判断します。
2.  **ドキュメント検証**: 現在のドキュメントに「やり方」が書かれているか？
    - ここで初めてドキュメントを開いて検索します。

### 4. ギャップ判定と分類 (Gap Analysis)

各事例を以下の 4 パターンに判定・分類します。

| 判定              | 実装 | ドキュメント | アクション                                       |
| :---------------- | :--- | :----------- | :----------------------------------------------- |
| **A. 記載済み**   | 〇   | 〇           | 問題なし（検証 OK）                              |
| **B. 追記が必要** | 〇   | ×            | **ドキュメントに追記**（「実はできます」を書く） |
| **C. 実装が必要** | ×    | -            | **機能追加を提案**（または「非対応」と明記）     |
| **D. 対象外**     | ×    | -            | プロダクトのスコープ外であることを確認           |

### 5. レポート作成と実行 (Reporting & Execution)

判定結果を以下の定型フォーマットでレポートし、ユーザーのアクション（追記 or 実装）を仰ぎます。

```
事例：[ユースケース名]
判定：[判定結果 A-D]
詳細：[理由・補足]（例：マクロ機能で擬似的に可能だが、ユーザーには伝わりにくい）
```

- **[Workflow-010]** 等と連携し、必要なタスク（ドキュメント修正、機能追加）を作成して実行します。

## チェックリスト

- [ ] ドキュメントを見ずにユースケースを洗い出したか？
- [ ] 「できること」だけでなく「やりたいこと」ベースで考えたか？
- [ ] 判定結果（特に B と C）に対して、具体的なネクストアクション（追記案など）を提示したか？

## 知見の種 (Seed) の記録

運用を通じて得られた「コーディングパターンの改善案」や「ルールの例外事例」は、`seeds.md` (Working Dir) へ記録してください（[Workflow-seed] 準拠）。
