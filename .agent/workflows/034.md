# [Workflow-034] 列挙項目の調査と表形式報告 (Enumeration & Reporting)

## 概要

ファイル、ルール、コード定義など、**「列挙可能な項目」** に対して調査や評価を行う際の手順と、その結果を **Markdown Table** で可視化するための標準ワークフローです。
調査対象が膨大な場合の戦略（グルーピング・境界値選定）も含みます。

## 適用条件 (Trigger)

以下のようなリクエストを受けた場合に適用します。

- 「〜の一覧を精査して」
- 「以下の項目について、A と B の観点でチェックして」
- 「〜における不整合がないかリストアップして」
- `@[.agent/workflows]` のようなディレクトリ指定での全体調査依頼

## 出力フォーマット (Output Requirement)

調査結果は **必ず Markdown Table 形式** で提示してください。
※ [Workflow-029] は UI 実装のルールであり、ドキュメントや報告でのテーブル利用を禁止するものではありません。

### テーブル構成例

| 項目 (Item) | 観点 A (Criteria A) | 観点 B (Criteria B) | 判定 (Status) | 備考 (Notes) |
| :---------- | :------------------ | :------------------ | :------------ | :----------- |
| `001.md`    | ✅ OK               | ⚠️ Low              | **Pass**      | -            |
| `002.md`    | ❌ NG               | ✅ High             | **Fix Req**   | 記述が古い   |

---

## 手順 (Step-by-Step)

### 1. 調査対象と観点の定義 (Define)

- **行 (Rows)**: 何を列挙するか（全ファイル、全関数、全変数など）。
- **列 (Columns)**: ユーザー指示に基づき、どのような基準で評価するか。
  - 必須列: 「項目名」「判定」「備考/根拠」

### 2. 調査戦略の決定 (Strategy)

対象件数に応じてアプローチを選択します。

#### A. 全数調査 (Small / Medium Scale)

- 件数が処理可能（〜20 件程度）な場合、全ての項目を一つずつ確認します。

#### B. 分類と代表値抽出 (Large Scale)

- 件数が膨大で全数調査が困難な場合、[Workflow-033] (思考シミュレーション) を応用し、**「機能的グループ」** に分類して代表値・境界値を調査します。
  - **カテゴリ化**: 機能や役割でグループ化（例：修飾キー群、文字キー群）。
  - **境界値チェック**: カテゴリ内で特異な挙動をしそうなもの（例：Enter, F1/F12）を重点的に確認。

### 3. テーブルの構築 (Construct Table)

- 調査結果を Markdown Table にまとめます。
- **重要**: テーブルはコードブロック ` ``` ` で囲まず、そのままレンダリングさせてください。
- 行数が多すぎる場合は、カテゴリごとにテーブルを分割することを検討してください。
